# ChronoMind Final System Report

## Architecture Summary

ChronoMind is a temporal memory retrieval system, not a chatbot. The system treats personal knowledge as an evolving graph of events and answers questions by combining:

- Qdrant vector retrieval
- Neo4j graph traversal
- BM25 lexical retrieval
- query understanding
- multi-stage ranking and reranking
- chronological context reconstruction
- LLM reasoning with deterministic fallback
- evaluation and observability

The production stack runs through Docker Compose with:

- FastAPI backend
- Next.js frontend
- Neo4j graph database
- Qdrant vector database

## Implemented Features

### Retrieval and Ranking

- Hybrid retrieval uses vector, graph, and BM25 channels independently.
- Query understanding classifies questions into decision trace, temporal evolution, causal inference, comparison, and factual recall.
- Graph traversal supports multi-hop expansion and uses causal, contradiction, refinement, and reinforcement edges.
- Ranking uses calibrated feature scoring with:
  - vector similarity
  - lexical score
  - graph distance
  - graph centrality
  - temporal distance
  - recency
  - event type weighting
  - causal edge strength
  - entity overlap
  - importance score
  - memory strength
- Diversity filtering reduces redundant results before context assembly.

### Knowledge Evolution

- Events now carry `importance_score` and `memory_strength`.
- Belief and opinion events are linked with:
  - `CONTRADICTS`
  - `REFINES`
  - `REINFORCES`
- Retrieval can surface belief evolution chains for questions like:
  - how has my opinion changed?
  - what did I used to believe?
  - when did my understanding evolve?

### Context Reconstruction

- Context is assembled chronologically, not as a raw top-k dump.
- Duplicate and low-value events are compressed out.
- Timeline order and causal ordering are preserved for LLM input.

### Observability

- Query debug trace includes:
  - query classification
  - candidate counts per retrieval channel
  - pre-fusion ranking
  - post-fusion ranking
  - reranked top results
  - final timeline context
- A retrieval inspector is exposed in the frontend.

### Frontend Productization

- Query answer view now shows:
  - answer
  - confidence
  - supporting events
  - retrieval inspector
- Timeline view shows chronological memory events with score badges.
- Graph view is interactive with zoom, pan, selection, and relationship inspection.
- A system dashboard shows core corpus statistics.

### Evaluation

- Added `evaluation/run_eval.py`.
- Metrics computed:
  - Recall@K
  - MRR
  - NDCG@10
  - Temporal Ordering Accuracy
  - Redundancy Score

## Issues Found and Fixed

- Missing stats endpoint and dashboard exposure.
- Frontend query view lacked retrieval trace inspection.
- Graph view was a static list instead of an interactive graph.
- Ingestion did not persist importance and memory strength fields.
- Belief evolution chains were not explicitly represented.
- Ranking did not account for memory importance and decay.
- Some frontend build dependencies were not installed in the local workspace.

## Unresolved Issues

- Local deterministic reasoning is still used when no LLM key is configured.
- The graph visualization is interactive but still uses a lightweight SVG layout rather than a physics engine.
- Demo data is synthetic and intentionally limited in scope.

## Recommended Future Work

- Add a proper graph layout engine for denser knowledge graphs.
- Persist retrieval feedback signals for stronger memory strength updates over time.
- Add more seeded life-event categories to broaden evaluation coverage.
- Upgrade the reasoning layer to support optional LLM-backed answer grounding when credentials are present.
